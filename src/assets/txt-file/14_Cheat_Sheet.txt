OOP QUICK REFERENCE
--------------------------------------------------------------------

FOUR PILLARS:
Encapsulation: private fields + public properties
Abstraction: interfaces and abstract classes hide complexity
Inheritance: class Child : Parent (single), class C : IA, IB (multiple interfaces)
Polymorphism: virtual/override, interface dispatch

INTERFACE vs ABSTRACT CLASS:
Interface: contract only (no state), multiple, always public, no constructor
Abstract: can have state, implementation, constructor; single inheritance

IS-A vs HAS-A:
Is-a → inheritance (Dog is an Animal)
Has-a → composition (Car has an Engine) — prefer composition!

WHEN TO USE WHICH:
interface: define a contract (IRepository, ILogger, IPaymentProcessor)
abstract class: shared implementation with variation (HttpMessageHandler, Stream)
sealed class: prevent inheritance (string in C# is sealed)
struct: small value type with no identity, few fields, rarely boxed

--------------------------------------------------------------------
SOLID QUICK REFERENCE
--------------------------------------------------------------------

S — Single Responsibility: one class, one reason to change
    Bad: OrderService saves to DB AND sends emails
    Good: OrderService + EmailService separately

O — Open/Closed: extend without modifying existing
    Bad: switch(type) to add new behavior
    Good: IStrategy interface, add new class

L — Liskov Substitution: subtype fully substitutable for base
    Bad: Square.Width setter also sets Height (breaks Rectangle assumption)
    Good: derived class strengthens not weakens base contract

I — Interface Segregation: small focused interfaces
    Bad: IWorker with Work() and Eat() — robots don't eat
    Good: IWorkable + IFeedable separately

D — Dependency Inversion: depend on abstractions
    Bad: new SmtpEmailSender() inside OrderService
    Good: inject IEmailSender via constructor

DESIGN PATTERNS QUICK REFERENCE
--------------------------------------------------------------------

CREATIONAL (object creation):
Singleton: one instance (Lazy<T> + Instance property)
Factory Method: subclass decides what to create (ILoggerFactory)
Abstract Factory: family of related objects (IUIFactory → DarkThemeFactory)
Builder: step-by-step complex construction (WebHostBuilder)
Prototype: clone existing object

STRUCTURAL (composing objects):
Adapter: incompatible → compatible interface (legacy API wrapper)
Decorator: add behavior without modifying (CachingRepository wraps Repository)
Facade: simple interface to complex subsystem (OrderFacade)
Composite: tree of objects (file system, UI component tree)
Proxy: surrogate — lazy loading, security, logging

BEHAVIORAL (object collaboration):
Strategy: swappable algorithms (IPaymentStrategy)
Observer: publish-subscribe (C# events, INotifyPropertyChanged)
Command: encapsulate request (MediatR ICommand, undo/redo)
Chain of Responsibility: pass request down a chain (middleware)
Template Method: algorithm skeleton, subclass fills steps
State: behavior changes with state (order state machine)

DRIVER DEVELOPMENT QUICK REFERENCE
--------------------------------------------------------------------

WINDOWS DRIVERS:
User Mode (Ring 3) → Kernel Mode (Ring 0)
I/O Request Packet (IRP) = data structure representing I/O
WDM (legacy) → KMDF (modern kernel) → UMDF (modern user-mode)
IRQL: PASSIVE_LEVEL(0) < APC_LEVEL(1) < DISPATCH_LEVEL(2) < DIRQL
Non-paged pool: accessible at any IRQL
Paged pool: only at PASSIVE/APC_LEVEL
Tools: WinDbg, DbgView, Driver Verifier, WPP Tracing

LINUX DRIVERS:
Everything is a file — devices in /dev
Character device: byte stream (keyboard, serial)
Block device: random access (disk)
Module: insmod driver.ko / rmmod driver / dmesg
file_operations: .open, .read, .write, .unlocked_ioctl
copy_to_user / copy_from_user — never dereference user pointers directly
kmalloc(size, GFP_KERNEL) — kernel heap allocation
printk(KERN_INFO "msg\n") — kernel logging

KEY DRIVER QUESTION: How does user mode talk to a driver?
Windows: CreateFile("\\\\.\\DeviceName") → DeviceIoControl(hDevice, controlCode, ...)
Linux: open("/dev/mydevice") → ioctl(fd, IOCTL_CODE, ...) or read()/write()

C++ QUICK REFERENCE
--------------------------------------------------------------------

MEMORY:
Stack: local variables, automatic cleanup, fixed size
Heap: new/delete, manual management, flexible lifetime

POINTERS:
int* p = &x; — pointer
*p = 5; — dereference
p = nullptr; — null pointer

SMART POINTERS (prefer over raw):
unique_ptr<T>: sole owner, auto-delete — make_unique<T>()
shared_ptr<T>: shared ownership, ref-count — make_shared<T>()
weak_ptr<T>: non-owning reference to shared_ptr object

RAII: resource acquired in constructor, released in destructor

REFERENCES:
int& ref = x; — alias, cannot be null, cannot rebind
void f(const string& s) — read-only reference parameter (no copy!)
void f(string&& s) — rvalue reference (move semantics)

std::move(x) — cast to rvalue (enable move)
std::forward<T>(x) — perfect forwarding in templates

C# → C++ MAPPING:
List<T> → std::vector<T>
Dictionary<K,V> → std::unordered_map<K,V>
SortedDictionary → std::map<K,V>
string → std::string
Func<T,R> → std::function<R(T)>
var x = ... → auto x = ...
foreach (x in c) → for (const auto& x : c)
IEnumerable<T> → range with begin()/end()
