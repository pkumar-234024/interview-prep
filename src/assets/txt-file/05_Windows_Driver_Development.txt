SECTION 5: WINDOWS DRIVER DEVELOPMENT BASICS
==================================================================

--------------------------------------------------------------------
BEGINNER'S GUIDE TO WINDOWS DRIVER DEVELOPMENT
--------------------------------------------------------------------

WHAT IS A DEVICE DRIVER?
A device driver is a software component that allows the operating system to communicate with hardware devices. Think of it as a translator between the OS and hardware.

Why Drivers Exist:
- Hardware diversity — thousands of devices, each with unique communication protocols
- OS abstraction — applications use a standard API (Win32); drivers handle hardware-specific details
- Privilege — hardware access requires kernel privileges unavailable to normal applications

KERNEL MODE vs USER MODE
----

User Mode (Ring 3):
- Normal applications run here (Chrome, Word, your C# apps)
- Limited access to hardware and OS resources
- Crash affects only that process
- Examples: Win32 applications, .NET programs, services

Kernel Mode (Ring 0):
- OS kernel, device drivers run here
- Full hardware access
- A bug can crash the entire system (Blue Screen of Death - BSOD)
- Extremely performance-sensitive

Transition: System calls move from user mode to kernel mode.
In C#: When you call File.Open(), it eventually crosses the boundary via P/Invoke → Win32 API → kernel.

WINDOWS DRIVER ARCHITECTURE
---

I/O Manager: Central coordinator for all I/O operations. Receives I/O requests from user mode, packages them as IRPs (I/O Request Packets), routes to appropriate driver.

Driver Stack: Multiple drivers layered vertically.
Application → Win32 API → I/O Manager → Filter Driver → Function Driver → Bus Driver → Hardware

Example (USB keyboard):
- USB HID Class Driver (function driver) — handles keyboard specifics
- USB Hub Driver — USB topology
- USB Host Controller Driver — physical USB

IRP (I/O Request Packet):
- Data structure that represents an I/O request
- Contains: operation type, buffer, parameters, completion routine
- Driver processes IRP and either completes it or passes to lower driver

WINDOWS DRIVER MODELS
----

WDM (Windows Driver Model) — Legacy:
- Low-level, direct IRP handling
- Complex, error-prone
- Still used in legacy hardware
- Three driver types: Bus, Function, Filter

KMDF (Kernel-Mode Driver Framework) — Recommended for Kernel Mode:
- Wrapper over WDM that handles common boilerplate
- Automatic IRP handling, PnP, power management
- Uses framework objects (WDFDEVICE, WDFQUEUE, WDFREQUEST)
- Much easier than raw WDM
- Use for: disk drivers, network adapters, low-latency devices

UMDF (User-Mode Driver Framework) — Recommended for User Mode:
- Runs in user mode — crashes don't BSOD the system
- Safer but slightly lower performance
- Communicates with kernel via kernel-mode stub
- Use for: USB devices, HID, many peripherals

DRIVER LIFECYCLE (Plug and Play)
----

1. AddDevice: Driver registered for device; create device object
2. StartDevice: Hardware resources allocated; device initialized
3. StopDevice: Device temporarily stopped (e.g., resource rebalancing)
4. RemoveDevice: Device unplugged; cleanup resources
5. UnloadDriver: Driver DLL unloaded from kernel memory

DRIVER COMMUNICATION
----

DeviceIoControl: Main mechanism for user-mode to driver communication.
User mode sends control code + input buffer → driver processes → returns output buffer.
Control codes defined with CTL_CODE macro.

Shared Memory: User-mode and kernel-mode share memory region.

Events/Callbacks: Driver signals user mode when event occurs.

DRIVER DEBUGGING BASICS
----

Tools:
- WinDbg: Kernel debugger; attach to live kernel or crash dump
- DbgView: Monitor DbgPrint() messages (like Console.WriteLine for drivers)
- Driver Verifier: Runtime validation tool — catches memory errors, deadlocks
- WPP Tracing: Efficient kernel tracing (like ETW for drivers)

Common BSOD Causes in Drivers:
- NULL pointer dereference at IRQL > PASSIVE_LEVEL
- Accessing paged memory at DISPATCH_LEVEL (IRQL issue)
- Stack overflow
- Double free

Tools Installation: Windows Driver Kit (WDK) from Microsoft.

IRQL (Interrupt Request Level):
- PASSIVE_LEVEL (0): Normal code, all memory accessible
- DISPATCH_LEVEL (2): Scheduler runs, no paged memory access
- DIRQL: Hardware interrupt handling

WINDOWS DRIVER INTERVIEW QUESTIONS (50)
--------------------------------------------------------------------

Q1. What is a device driver?
Answer: A software component acting as a translator between the OS and hardware, enabling standardized communication.

Q2. What is the difference between kernel mode and user mode?
Answer: Kernel mode (Ring 0) has full hardware access; bugs crash the system. User mode (Ring 3) is isolated; crashes don't affect other processes. Applications run in user mode; drivers typically in kernel mode.

Q3. What is an IRP?
Answer: I/O Request Packet — the data structure representing an I/O operation. Contains major function code, parameters, and buffers. Drivers process IRPs and either complete them or pass down the driver stack.

Q4. What is the I/O Manager in Windows?
Answer: The OS component that coordinates all I/O operations. Receives user-mode I/O requests, creates IRPs, routes to appropriate driver, and manages completion.

Q5. What are the three types of WDM drivers?
Answer: Bus driver (manages physical/virtual bus, discovers child devices), Function driver (main driver for a device, provides device functionality), Filter driver (optionally modifies behavior above or below function driver).

Q6. What is WDM?
Answer: Windows Driver Model — the legacy kernel-mode driver framework. Requires direct IRP handling, manual PnP/power management. Powerful but complex and error-prone.

Q7. What is KMDF?
Answer: Kernel-Mode Driver Framework — builds on WDM, provides object model (WDFDEVICE, WDFQUEUE). Automates PnP, power management, synchronization. Significantly reduces driver code complexity.

Q8. What is UMDF?
Answer: User-Mode Driver Framework — drivers run as user-mode processes. Safer (crashes don't BSOD), easier debugging. Suitable for most non-performance-critical devices.

Q9. What is DeviceIoControl?
Answer: Win32 API function for user-mode to send custom I/O control codes to a driver. Passes input/output buffers. Driver handles IOCTL_* codes in its IRP_MJ_DEVICE_CONTROL handler.

Q10. What is a BSOD and what causes it in drivers?
Answer: Blue Screen of Death — system crash dump when kernel encounters unrecoverable error. Common driver causes: NULL pointer dereference, accessing paged pool at DISPATCH_LEVEL, deadlocks, stack overflow.

Q11. What is IRQL?
Answer: Interrupt Request Level — the priority at which code runs in the kernel. PASSIVE_LEVEL (0) for normal code, APC_LEVEL (1), DISPATCH_LEVEL (2) for scheduler/DPC, DIRQL for hardware interrupts. Code at high IRQL cannot access paged memory.

Q12. What is paged vs non-paged pool?
Answer: Non-paged pool: memory always physically present — safe to access at any IRQL. Paged pool: may be paged to disk — only accessible at PASSIVE or APC_LEVEL. Driver data structures accessed at high IRQL must be in non-paged pool.

Q13. What is a DPC (Deferred Procedure Call)?
Answer: A mechanism to defer interrupt processing work to DISPATCH_LEVEL (lower than hardware interrupt). ISR (Interrupt Service Routine) does minimal work, schedules DPC for remaining processing. Reduces time spent at high IRQL.

Q14. What is a device object?
Answer: A DEVICE_OBJECT structure representing a logical device in the kernel. Each driver creates one via IoCreateDevice(). Applications open a handle to the device via CreateFile("\\\\.\\MyDevice").

Q15. What is a driver object?
Answer: DRIVER_OBJECT structure representing the loaded driver itself. Created by the OS when driver loads. Contains dispatch function table (IRP handlers), DriverUnload routine.

Q16. What is DriverEntry?
Answer: The entry point function for a kernel-mode driver — equivalent to main(). Receives DRIVER_OBJECT and registry path. Registers dispatch routines and AddDevice. Called once when driver loads.

Q17. What is AddDevice in KMDF?
Answer: Called when PnP manager finds a device the driver should handle. Driver creates WDFDEVICE object here. Set up queues, timers, DMA. KMDF's equivalent of WDM AddDevice.

Q18. What is a symbolic link in driver context?
Answer: Maps a device object to a user-accessible name: IoCreateSymbolicLink creates "\\DosDevices\\MyDevice" → "\\Device\\MyDevice". User mode opens CreateFile("\\\\.\\MyDevice") — OS resolves via symbolic link.

Q19. What is PnP (Plug and Play)?
Answer: Windows mechanism for device hot-plug. Drivers must handle PnP IRPs: IRP_MN_START_DEVICE, IRP_MN_STOP_DEVICE, IRP_MN_REMOVE_DEVICE, IRP_MN_QUERY_CAPABILITIES. KMDF handles most PnP automatically.

Q20. What is power management in drivers?
Answer: Drivers handle power state transitions (S0-S5 system states, D0-D3 device states). IRPs: IRP_MN_SET_POWER, IRP_MN_QUERY_POWER. KMDF automates power management via WdfDeviceInitSetPowerPolicyOwnership.

Q21. What is a filter driver?
Answer: Sits above or below the function driver in the device stack. Intercepts IRPs to modify behavior: add logging, encryption, filtering. Example: anti-virus file system filter driver intercepts all file I/O.

Q22. What is the WDK?
Answer: Windows Driver Kit — SDK for developing Windows drivers. Includes headers, libraries, build tools, samples, documentation. Download from Microsoft's developer site. Required for driver compilation.

Q23. What is WPP tracing?
Answer: Windows Software Trace Preprocessor — efficient kernel tracing mechanism. Define trace messages at build time; at runtime only enabled traces are active. Use TraceView or logman to collect. Replaces DbgPrint for production tracing.

Q24. What is Driver Verifier?
Answer: Windows tool that adds runtime checks to drivers: buffer overrun detection, IRQL violation detection, deadlock detection, memory leak detection. Enable for target driver during testing; not for production (adds overhead, may crash intentionally on error).

Q25. What is a spinlock?
Answer: A synchronization primitive for kernel-mode multi-processor use at DISPATCH_LEVEL. Thread busy-waits (spins) rather than sleeping. KeAcquireSpinLock()/KeReleaseSpinLock(). Must be held briefly; no sleeping while held.

Q26. What is a fast mutex?
Answer: Kernel synchronization for PASSIVE_LEVEL (allows sleeping, unlike spinlock). Used in drivers for PASSIVE_LEVEL critical sections. ExAcquireFastMutex()/ExReleaseFastMutex(). Not recursive.

Q27. What is DMA in driver context?
Answer: Direct Memory Access — hardware transfers data directly to/from RAM without CPU. Driver sets up DMA adapter, allocates common buffer or uses scatter/gather. KMDF provides WdfDmaEnabler, WdfDmaTransaction.

Q28. What is a minidriver?
Answer: A driver that implements device-specific functionality while relying on a class driver for common protocol handling. Example: USB HID minidriver for custom HID devices; relies on HidClass class driver.

Q29. What is the Hardware Abstraction Layer (HAL)?
Answer: HAL abstracts hardware differences between processor architectures and platform variations. Provides APIs for I/O ports, timers, bus interfaces. Drivers use HAL APIs rather than direct hardware registers.

Q30. What is kernel debugging setup?
Answer: Connect two machines via serial, USB, network, or FireWire (1394). Target: enable kernel debugging (bcdedit /debug on). Host: WinDbg.exe connects to target. Set symbols path to Microsoft Symbol Server. Break on driver load: sxe ld:mydriver.sys.

Q31. What is the difference between synchronous and asynchronous I/O in driver context?
Answer: Synchronous: driver completes IRP before returning. Asynchronous: driver marks IRP pending (IoMarkIrpPending), returns STATUS_PENDING, completes IRP later via IoCompleteRequest(). KMDF manages this via queue and request objects.

Q32. What is a kernel event (KEVENT)?
Answer: Kernel synchronization object. Two types: notification (remains signaled until manually reset), synchronization (auto-resets after one thread is released). Use KeInitializeEvent, KeSetEvent, KeWaitForSingleObject. Like System.Threading.ManualResetEvent/AutoResetEvent in .NET.

Q33. What is a callback routine in KMDF?
Answer: KMDF uses callback model instead of IRPs. Register callbacks like EvtIoRead, EvtIoWrite, EvtIoDeviceControl for request handling. EvtDevicePrepareHardware for resource allocation. Framework calls your callbacks appropriately.

Q34. What is a queue in KMDF?
Answer: WDF_IO_QUEUE — manages incoming I/O requests (WDFREQUESTs). Types: sequential (one at a time), parallel (all at once), manual (driver controls dispatch). Default queue handles all requests; add additional queues for specific request types.

Q35. What is a WDFREQUEST?
Answer: KMDF object wrapping an IRP. Access request parameters, buffers. Complete with WdfRequestComplete(). Forward down stack with WdfRequestSend(). KMDF handles memory management of WDFREQUESTs automatically.

Q36. What are memory types in Windows kernel?
Answer: Paged pool (kernel heap that can be paged), NonPaged pool (always resident), Lookaside lists (pre-allocated blocks), MmAllocateContiguousMemory (physically contiguous). ExAllocatePoolWithTag for dynamic allocation. Tag helps identify allocations in debugging tools.

Q37. What is a buffer method for IOCTL?
Answer: How input/output data is transferred: METHOD_BUFFERED (OS copies to/from system buffer — safe, easy), METHOD_IN_DIRECT/METHOD_OUT_DIRECT (MDL-mapped user buffer), METHOD_NEITHER (driver accesses user buffer directly — risky, need probe/lock). Beginners use METHOD_BUFFERED.

Q38. What is MDL (Memory Descriptor List)?
Answer: Describes a virtually contiguous memory region with its physical page layout. Used for locking user-mode buffer pages so driver can safely access during I/O. MmBuildMdlForNonPagedPool, MmProbeAndLockPages.

Q39. What is OBJ_KERNEL_HANDLE?
Answer: Attribute for kernel-created object handles — makes handle accessible only from kernel mode. Prevents user-mode code from accidentally or maliciously accessing kernel objects via the same handle.

Q40. What is IoCompleteRequest?
Answer: Called to complete an IRP after processing. Signals I/O Manager that request is done. Sets IoStatus.Status and IoStatus.Information. Specifies priority boost for waiting thread. Must be called exactly once per IRP.

Q41. What is a class driver vs minidriver?
Answer: Class driver: handles common protocol (USB HID class, SCSI class). Minidriver: device-specific customization. Microsoft provides class drivers; ISVs write minidrivers. Reduces code duplication; ensures protocol compliance.

Q42. What is SetupAPI?
Answer: User-mode API for device installation and management. Used in driver installers. Enumerates devices, installs INF files, manages device classes. Wrapped by Device Installation APIs.

Q43. What is an INF file?
Answer: Text file describing driver installation — lists files to copy, registry keys to set, device IDs the driver supports. Required for Windows to install a driver. Digitally signed INF+driver package required for production.

Q44. What is driver signing?
Answer: Microsoft requires drivers to be digitally signed (Authenticode). Test signing mode allows unsigned drivers for development. Production: Extended Validation (EV) certificate + Microsoft's Hardware Lab Kit (HLK) certification for Windows Logo.

Q45. What is a kernel-mode rootkit?
Answer: Malicious driver that hides itself and other malware. Uses driver privileges to intercept system calls, hide files, processes. Demonstrates why driver signing and Secure Boot are critical security measures.

Q46. What is Device Tree in Windows?
Answer: Hierarchical representation of all hardware. Root is ACPI/hardware platform. Each node is a PDO (Physical Device Object) created by bus driver. Function driver adds FDO (Functional Device Object) on top. Viewable in Device Manager.

Q47. What is WDF object hierarchy?
Answer: WDFDRIVER (root) → WDFDEVICE → WDFQUEUE, WDFMEMORY, WDFTIMER, WDFWORKITEM. Each object has a parent; when parent is deleted, children are automatically deleted. Simplifies resource cleanup.

Q48. What is a work item in KMDF?
Answer: WDF work items allow scheduling work at PASSIVE_LEVEL from DISPATCH_LEVEL contexts. WdfWorkItemCreate + WdfWorkItemEnqueue → callback runs at PASSIVE_LEVEL. Like C# Task.Run() from an interrupt handler.

Q49. What is ETW (Event Tracing for Windows)?
Answer: High-performance kernel-level tracing infrastructure. Used by WPP tracing, Windows Performance Analyzer, Process Monitor. Circular buffer in kernel — very low overhead. Collect with xperf or WPR; analyze with WPA.

Q50. How would you approach writing your first kernel driver?
Answer: Start with WDK samples (kmdf\toaster or kmdf\echo). Set up WDK + Visual Studio. Build and install echo sample. Add DbgPrint statements; monitor with DebugView. Add simple IOCTL for user-mode communication. Test with Driver Verifier enabled. Graduate to real hardware with datasheet.
